Class RabbitWorkerQueue

  • java.lang.Object
    • com.hpe.caf.worker.queue.rabbit.RabbitWorkerQueue
  • All Implemented Interfaces:
    HealthReporter, ManagedWorkerQueue, WorkerQueue

    public final class RabbitWorkerQueue
extends Object
implements ManagedWorkerQueue
    This implementation uses a separate thread for a consumer and producer, each with their own Channel. These threads handle operations via a BlockingQueue of Event objects. In all scenarios where the tasks triggered by the message take significantly longer than the handling of the messages themselves (which should hopefully be true of all microservices), this implementation should work. This implementation has three routing keys, assumed to be on a direct exchange (hence effectively being queue names). There is the input queue to receive messages from, the retry queue (which may be the input queue) where redelivered messages get republished to, and the rejected queue which is where messages that could not be handled are put. There are an unlimited number of possible output queues as defined by the Worker's response. All published messages use RabbitMQ confirmations.

      • start

        public void start(TaskCallback callback)
           throws QueueException
        Open queues to start accepting tasks and results. Create a RabbitMQ connection, and separate incoming and outgoing channels. The connection and channels are managed by Lyra, so will attempt to re-establish should they drop. Declare the queues on the appropriate channels and kick off the publisher and consumer threads to handle messages. Since this code uses publisher confirms, it is important currently to declare the publisher channel before the consumer channel, otherwise during a connection drop scenario, Lyra can report the publish sequence number for the "old" channel before recovering it.
